# Vindkraft AR Katrineholm

A Swedish-language progressive web app that lets people in Katrineholm point their phone camera at the horizon and see, in augmented reality, the 29 wind turbines planned at Länsterberget — with live GPS/compass-based placement, distance labels, a map view, and a petition button tied to the town's real wind-power referendum debate.

## Run & Operate

- `pnpm --filter @workspace/vindkraft-ar-katrineholm run dev` — run the app (dev workflow: "artifacts/vindkraft-ar-katrineholm: web")
- `pnpm --filter @workspace/vindkraft-ar-katrineholm run typecheck` — typecheck this artifact
- `pnpm run typecheck` — full typecheck across all packages
- No database, no backend, no OpenAPI codegen — this artifact is fully client-side.

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- React + Vite artifact (`artifacts/vindkraft-ar-katrineholm`)
- Three.js for the AR/3D rendering layer
- proj4 for SWEREF99 TM (EPSG:3006) → WGS84 coordinate conversion
- vite-plugin-pwa for installable PWA support (manifest + service worker)
- Native browser APIs: `getUserMedia` (camera), `navigator.geolocation` (GPS), `DeviceOrientationEvent` (compass)
- No backend/database for this artifact — signatures for the petition are stored in `localStorage`.

## Where things live

- `src/lib/turbines.ts` — the 29 turbine records (SWEREF99 TM easting/northing + height), grounded near the real Länsterberget wind farm site north of Katrineholm.
- `src/lib/sweref.ts` — proj4-based SWEREF99 TM ↔ WGS84 conversion.
- `src/lib/geo.ts` — haversine distance, bearing, angle normalization, night-time check, Swedish distance formatting.
- `src/hooks/useGeolocation.ts`, `useCompassHeading.ts`, `useCameraStream.ts`, `useWindSound.ts` — device sensor/media hooks.
- `src/hooks/useDeviceOrientation.ts` — adaptive ("one-euro"-style) two-rate EMA compass/tilt smoothing: small raw heading deltas (<3°, likely magnetometer noise) get heavy damping (tau=0.9s), large deltas (>12°, real turns) get fast response (tau=0.12s), linearly interpolated in between — a plain fixed-tau EMA was too responsive to filter out several-degrees-even-when-stationary magnetometer noise. Also rejects single readings implying >720°/s turn rate as sensor glitches. Listens to both `deviceorientationabsolute` and plain `deviceorientation` (needed since iOS never fires the former), but once any absolute-referenced reading arrives (event type `deviceorientationabsolute`, `event.absolute === true`, or iOS's `webkitCompassHeading`) it latches on and ignores further non-absolute `deviceorientation` events — otherwise two conflicting heading sources (some browsers' plain `deviceorientation.alpha` isn't reliably north-referenced) feed the same smoother and cause large, unattenuated bearing swings (heading error maps directly to on-screen angle, unlike GPS position error which is angularly damped by distance to the turbines). Exposes `hasSettled` (true once heading stability holds ≥0.75 for a continuous 1200ms after `hasFix`, or after a 5000ms max-wait fallback) — `Home.tsx`'s `ready` gate requires it so turbines don't render against a still-settling compass, with a "Kalibrerar kompass" countdown shown meanwhile. Also owns the two-step pre-AR compass calibration exposed as `calibrationPhase` (`"flat" | "vertical" | "done"`) and `calibrationProgress`: `startCalibrationTracking()` resets to `"flat"`, and the raw (unsmoothed) heading is bucketed into 12 sectors, gated on the device's pitch (`beta`) actually matching the active phase (`isFlatBeta`/`isVerticalBeta`, ±45° tolerance around 0°/180° for "flat" and 90° for "vertical") — sweeping 5/12 sectors while flat completes step 1 and advances to `"vertical"`, sweeping 5/12 more while upright completes step 2 (`"done"`). This drives `LoadingSequence.tsx`'s two-substep calibration UI.
- `src/hooks/useSkyDetection.ts` — lightweight canvas-2D (no WebGL/ML) brightness/texture/saturation heuristic on a 12x8 grid of the camera frame; drives both `outdoorConfidence`/`indoors` and (via `isPointSky`/`getOcclusionGrid`) the real per-pixel sky/occlusion mask consumed by `ARScene.tsx`'s shader. `method` is permanently `"disabled"` — the previous heavier ML segmentation model was removed for causing real-device freezes (see Gotchas) and is not reintroduced by this heuristic. `classifyCell` also has a texture-independent `veryBrightAndDesaturated` (avgLum>160, sat<0.15) OR-branch so bright textured clouds always classify as sky — otherwise clouds could get misclassified as "occluded" and false-tint turbines behind them.
- `src/components/ARScene.tsx` — the Three.js AR overlay (turbine meshes, name/distance labels, blinking red aviation light at night, sun-synced lighting, Skuggflimmer blade-shadow flicker, exposes its canvas via `canvasRef` for Fotomontage capture). `layoutObjects()` grounds each turbine's world Y position with a real trig height-angle (turbine `groundHeightMeters` above sea level minus an assumed ~60m Katrineholm-town ground elevation and a ~1.6m eye height, over the real GPS distance — flat-terrain approximation, no user altitude is available), instead of a fixed pixel offset; object scale is likewise derived from real distance (`planeDist / (realDist * METERS_TO_UNITS)`) so it subtends the true angular size, with "förstärkt visning" only boosting scale upward from the grounded base, never lifting it. A safety clamp recomputes the elevation angle using a capped minimum distance if a turbine's top would otherwise exceed ~10° above the horizon (guards against a GPS-glitch collapsing the distance; not expected to trigger at this site's real turbine distances of several km).
- `src/components/CameraBackground.tsx`, `MapView.tsx`, `PetitionModal.tsx`, `PermissionGate.tsx` — supporting UI. `CameraBackground` accepts an optional `videoRef` to expose its `<video>` element for Fotomontage capture.
- `src/components/VisualizationControls.tsx` — sun/scale/visibility/night mode toggles, plus the Skuggflimmer toggle + info tooltip.
- `src/components/SoundLevelPanel.tsx` — live "🔊 Beräknad ljudnivå" dBA estimate panel (`SoundLevelBadge` shows "Väntar på GPS…" fallback until a position fix exists).
- `src/components/PhotoMontageModal.tsx` — Fotomontage capture preview with Spara/Ta ny bild/Dela (Web Share API) actions.
- `src/lib/soundLevel.ts` — dBA estimation from GPS distance to every turbine, logarithmic combination, severity color coding, exact disclaimer text.
- `src/lib/visualizationTypes.ts` — shared visualization mode types, incl. `shadowFlickerActive()` gating (only active in "current"/"low" sun modes).
- `src/lib/noiseImpact.ts` — weighted green/yellow/red "infraljud-/bullerpåverkan" score combining dBA level, contributing-turbine count, downwind wind direction (if available), and exposure duration; owns the exact Swedish disclaimer text (`NOISE_IMPACT_DISCLAIMER`) — do not reword it, must keep "kan bidra till"/"kan upplevas"/"för känsliga personer" phrasing and avoid absolute claims.
- `src/hooks/useWindDirection.ts` — fetches current wind direction/speed for the user's GPS position from the free Open-Meteo API (no key required); fails silently to `null` on network errors so the noise monitor just ignores the wind factor.
- `src/components/NoiseImpactMonitor.tsx` — `NoiseImpactBadge` (always-visible top-bar status) + `NoiseImpactPanel` (expandable detail panel with reasons + disclaimer), mirrors the `SoundLevelPanel.tsx` pattern. Both `SoundLevelPanel` and `NoiseImpactPanel` default to a compact view (headline number/level + independent "Visa mer" toggle) and only render distance/count/disclaimer detail when expanded.
- `src/hooks/useOutdoorConfidenceIndex.ts` — weighted "Outdoor Confidence Index" (0-100%: camera/AI sky-detection 45%, GPS accuracy 20%, ambient light 15%, compass stability 10%, motion/gyro 5%, WiFi/indoor signal 5%) with four tiers (`show` ≥90, `cautious` ≥70, `aim` ≥40, `hide` <40) driving turbine visibility gating in `Home.tsx`/`ARScene.tsx`.
- `src/hooks/useStableGeoPosition.ts` — freezes the lat/lon fed into the dBA calculation until the user moves ≥15m, so ordinary GPS jitter doesn't recompute sound level.
- `src/components/LoadingSequence.tsx` — branded ("Vindkraftsparken" heading, Katrineholm FRAMÅT orange accent) three-phase sequence shown immediately after "Starta visualisering" is pressed: (1) **calibration** — a real, sensor-gated two-substep compass calibration ("1. Liggande telefon" then "2. Stående telefon", each with its own instruction, spinning phone icon, progress bar and green checkmark on completion, driven by `useDeviceOrientation.ts`'s `calibrationPhase`/`calibrationProgress`; skipped entirely via the `skipCalibration` prop if orientation is unsupported or permission was denied) — followed by "Kompass kalibrerad ✓"; (2) a fixed **3-2-1 countdown** with a per-second Swedish status message (GPS → position → turbine placement/AR scene); (3) a **checklist** of 7 items ticking off on a 220ms interval (within the 150–300ms product spec). A per-substep hint (after 7s) and hard max-wait (after 18s) keep calibration from hanging forever on a sensor-less device — same watchdog pattern as `useCameraStream.ts`/`useGeolocation.ts`; the hint is now paired with a "Fortsätt ändå →" button that jumps straight to the countdown phase, giving the user an explicit manual override instead of only a timer to wait out. The whole sequence still runs independent of (in parallel with, not blocking on) the real GPS/compass/camera permission flow — if the real `ready` state isn't true yet once it finishes, the pre-existing sensor-waiting overlay (`!ready` block in `Home.tsx`, with GPS/compass-specific Swedish hint text) takes over exactly as before, so this is a purely presentational front-load and never delays or masks a real permission error.
- `src/hooks/useSmoothedGeoPosition.ts` — continuous time-based EMA low-pass filter (~2.5s base time constant, same pattern as `useDeviceOrientation.ts`'s compass smoothing) on raw `lat`/`lon`, feeding `ARScene`'s turbine placement (`userLat`/`userLon` in `Home.tsx`). The effective time constant scales up further when `geo.accuracy` is poor, and single implausible GPS "spike" readings (would imply a faster-than-walking jump) are rejected outright rather than blended in. Distinct from `useStableGeoPosition`: it smooths continuously rather than freezing in 15m jumps, so turbines still track real walking movement smoothly but no longer visibly "fladdrar" (flicker/jitter) from ordinary meter-scale consumer GPS noise, which previously fed straight into `ARScene.tsx`'s `layoutObjects()` on every >~11cm raw position change. Note: at the app's real-world viewing distances (turbines are several km from Katrineholm town), GPS noise alone produces well under 1° of bearing error — see `useDeviceOrientation.ts` below for the dominant, higher-impact jitter source that was fixed alongside this.
- `src/hooks/useSmoothedDba.ts` — 7s rolling average of the raw dBA estimate, throttled to at most one visible update per second; both `SoundLevelPanel`/`NoiseImpactMonitor` display and the wind sound's `dbaToGain` volume consume this same smoothed value so they always agree.
- `src/pages/Home.tsx` — wires camera + AR + GPS + compass + permission gate + top/bottom UI chrome + dBA panel + noise impact monitor (incl. exposure-duration timer) + Fotomontage capture + Outdoor Confidence Index gating together. Also reads a custom turbine placement (if any) via `AR_HANDOFF_KEY`/`loadCustomPlacement()` and swaps it in for the default 29-turbine set as `activeTurbines`, with a bottom-bar link to `/placera` and a button to clear the custom placement. The `!ready` waiting overlay shows a live status panel (📷 Kamera / 📍 GPS / 🧭 Kompass / 🌬️ AR-scen, each ✅/⏳/❌) plus always-visible "🧭 Kalibrera om riktning" (re-runs `orientation.startCalibrationTracking()` + `calibrateHorizon()`) and "📍 Uppdatera position" (`geo.retry()`) buttons so a user can self-unstick GPS/compass without reloading. A separate `manualContinue` override — surfaced as a "Fortsätt ändå →" button after `MANUAL_CONTINUE_ANYWAY_MS` (12s), only once camera+GPS both have data — lets the user force through a stuck-compass case (bypasses `orientation.hasFix`/`hasSettled` in the `ready` formula) but never bypasses the camera/GPS requirements themselves, since there'd be nothing to render AR against without them.
- `src/lib/ericsbergArea.ts` — Ericsberg placement-area boundary polygon (`ERICSBERG_BOUNDARY`, small, used for the outside-boundary warning), a separate much larger illustrative `ERICSBERG_ESTATE_AREA` polygon ("Ericsbergs mark", spanning roughly Forssjö/Stora Malm/Ericsberg/Djulöfors/Strångsjö-Norrlunda), `KOMMUN_POPULATION` (35,000 total; ~25,000 inner Katrineholm, rest in kransorterna), ~14 `HOUSEHOLD_CLUSTERS` covering Katrineholm centrum + kransorterna (Julita, Sköldinge, Strängsjö/Björkvik, Bie, etc. — not just the Ericsberg estate itself, so a turbine placed near central Katrineholm actually scores against real household density), sensitive/positive zones, and the "Placera vindkraftverken själv" disclaimer text. Also owns the boundary-override system: `getActiveBoundary()` returns a `localStorage`-persisted custom boundary (key `vindkraft-ar-katrineholm:customBoundary`) if one was saved via `setCustomBoundary()`, else the built-in `ERICSBERG_BOUNDARY` — `placementScoring.ts` and `PlacementMap.tsx` always read the boundary through this function (never the raw constant) so the boundary editor in `PlaceTurbines.tsx` actually affects scoring/rendering. `hasCustomBoundary()`/`resetCustomBoundary()` and `boundaryToGeoJson()` (closes the ring, `[lon,lat]` pairs) round out the editor's save/reset/export.
- `src/lib/placementScoring.ts` — `scorePlacement()`: weighted 0–100 impact score (distance to households, sensitive-zone proximity, spacing, "Skuggpåverkan vid låg sol", etc.) plus playful Swedish warnings for the placement game; also reused per-turbine (`scorePlacement([t])`) to color-code individual markers on the map. Also returns `turbineContributions: {id,score}[]` (sorted desc), consumed by `PlaceTurbines.tsx`'s "🐞 Felsökning (poäng)" debug panel (total score, nearest household/urban distance, factor breakdown, per-turbine contributions) — added so a strong-red score near central Katrineholm can be diagnosed by which turbine/factor is actually driving it, rather than guessed at.
- `src/lib/webMercatorTiles.ts` — shared Web Mercator tile/projection math (Esri World Imagery tiles, `computeTileLayout`, `fitBoundsToAspect`, `makeProjector`); used by both `MapView.tsx` and `PlacementMap.tsx` so the two maps stay visually consistent.
- `src/components/PlacementMap.tsx` — the free "kartverktyg" map: pannable (drag empty area), zoomable (wheel + pinch + double-tap, via a dynamic center/lat-span `ViewState` re-projected through `boundsFromView`/`fitBoundsToAspect` on every pan/zoom), Google-Maps-style gesture behavior when no turbine is selected. Tapping a turbine opens a small context menu (Flytta/Ta bort/Information/Avbryt) instead of drag-to-move; "Flytta" enters an explicit move-mode (top banner) where the *next* background tap commits the new position (`onMove`) and shows a brief fading old→new arrow/marker (`moveAnim`, ~900ms) — background pan/zoom/double-tap-zoom and empty-tap-to-add (`onAdd`) all work normally whenever no menu/move-mode is active. Single-click-to-add on empty space is debounced by `DOUBLE_TAP_WINDOW_MS` so it can be superseded by a genuine double-tap-to-zoom. Marker color is driven by a separate `colorTurbines` prop (not live `turbines`), memoized per-id in `colorById` — this is deliberately a *lagged* snapshot (see `PlaceTurbines.tsx`) so recomputing per-turbine `scorePlacement([t])` never runs on every pan/zoom-driven re-render (this, plus the parent's debounce, was the fix for pan/zoom crashes with many objects). Renders the `ERICSBERG_ESTATE_AREA` dark-fill/black-border "Ericsbergs mark" polygon when `showEstateBoundary` is true, and an always-visible 🏰 "Ericsbergs slott" marker at `ERICSBERG_CENTER` that toggles a popup ("Markägare: Caroline Bonde" — text-only placeholder, no logo asset was available). Uses a real (non-passive) native `wheel` listener via `useEffect` rather than React's `onWheel` so `preventDefault()` doesn't warn. An optional `boundaryDebugMode` prop (toggled via `PlaceTurbines.tsx`'s "🔢 Visa gränspunkter" button) overlays numbered lat/lon vertex markers for both the active boundary (orange) and `ERICSBERG_ESTATE_AREA` (cyan), for verifying polygon accuracy against the real map — only shown when `boundaryEditMode` is off. A separate `boundaryEditMode` prop drives the real interactive boundary editor: draggable vertex handles (`onVertexDrag`, pointer-capture drag via a `vertexDragRef` parallel to the existing pan/pinch refs), a small ✕ per vertex to remove it (`onVertexRemove`, only shown when >3 vertices remain so the polygon can't collapse), and a background tap in edit mode calls `onVertexAdd` instead of placing a turbine — inserted onto the nearest boundary edge (closest point on any segment, not just nearest vertex) so an arbitrary tap can't self-intersect the polygon. Renders from an `editableBoundary` prop (the parent's live in-progress edit) rather than `getActiveBoundary()` while `boundaryEditMode` is on, so edits are visible immediately without touching the persisted/scored boundary until explicitly saved.
- `src/pages/PlaceTurbines.tsx` (route `/placera`) — "Placera vindkraftverken själv" kartverktyg: starts from the 8 *real* planned turbines nearest Katrineholm (real WGS84 coords derived from `src/lib/turbines.ts`, not placeholder/fictional positions) but is an unlimited, growable list (click-to-add/remove on the map), live score/warnings panel, a "🗺️ Visa Ericsbergs mark" toggle for the wider estate overlay, save/compare placements (`localStorage` keys `vindkraft-ar-katrineholm:savedPlacements` and `vindkraft-ar-katrineholm:customPlacement`), and a "Se denna placering i AR" handoff back to the main AR view. Reachable with zero camera/GPS/compass permissions (see `PermissionGate.tsx`). Maintains two turbine snapshots: `turbines` (live, updates positions immediately on move/add/remove so the map feels responsive) and `committedTurbines` (used for `scorePlacement()` — household count/score/colors), which only jumps to match `turbines` after a `RECOMPUTE_DELAY_MS` (700ms) debounce, during which a "Beräknar påverkan…" pill is shown — this is both the smooth-recompute UX and the perf fix that keeps the O(n²)-ish full-placement scoring off the synchronous input path. A "✏️ Redigera gräns" toggle enters the boundary editor: an `editableBoundary` state (initialized from `getActiveBoundary()`) is drawn live on the map via `PlacementMap`'s `boundaryEditMode`/`editableBoundary` props and mutated by drag/remove/add handlers, so the polygon updates instantly while editing but never affects the actual score/persisted boundary until "💾 Spara gräns" calls `setCustomBoundary()` — a `boundaryVersion` counter is bumped alongside save/reset to force `result = useMemo(() => scorePlacement(...), [committedTurbines, boundaryVersion])` to re-run (`scorePlacement()` reads the boundary via the module-level `getActiveBoundary()`, not as an argument, so it wouldn't otherwise know to recompute). "📤 Exportera GeoJSON" builds a `Blob` from `boundaryToGeoJson()` and opens it via `window.open(url, "_blank")` rather than an `<a download>` link — see Gotchas. The "🐞 Felsökning (poäng)" debug panel shows `layerCounts` (household clusters/nature/cultural/water zone counts) plus a full per-turbine breakdown (`turbineDebug`, sorted by score) — nearest household/nature/cultural/water name+distance, household counts within 1/2/3 km, distance to Katrineholm centrum, noise dBA, shadow-flicker/visual scores, total score, and an outside-boundary warning per turbine.

## Architecture decisions

- Built as a plain React+Vite PWA (not Expo), per explicit user requirement — overrides the default mobile-artifact assumption.
- No backend: turbine data is static/client-side, and petition "signatures" are stored in `localStorage` only (no server persistence).
- AR is done without WebXR — turbines are positioned in a Three.js scene using bearing/distance relative to the user's GPS position and compass heading, for broad mobile browser compatibility.
- The 29 turbines are fictional but grounded in the real Länsterberget wind farm project (OX2) north of Katrineholm and the real 2022 local referendum on wind power.

## Product

- Camera-based AR view showing 29 wind turbines at true bearing/distance from the user, each labeled with name + distance.
- Blinking red aviation obstruction lights on turbines at night (22:00–06:00).
- Optional procedurally generated wind sound (Web Audio, no audio files), always routed through the device's main speaker (never the earpiece) even on iOS Safari.
- Live "🔊 Beräknad ljudnivå" dBA panel estimating sound exposure from turbine distance. The wind sound's actual playback volume scales continuously with this same dBA estimate (`soundLevel.ts`'s `dbaToGain`), instead of a plain on/off.
- Explicit always-visible "🔊 Ljud ute / 🔈 Ljud inne" toggle (top controls) that the user controls directly — always starts on "Ljud ute" at app launch. This toggle (not the camera-based sky heuristic) drives both the displayed dBA number (via `applyIndoorAttenuation`, a -35dB subtraction before the panel's own `dbaToGain`) and the actual wind sound gain — but the two use separate mechanisms by design: the panel number goes through `applyIndoorAttenuation` → `dbaToGain`, while the actual playback gain is computed as `applyIndoorGain(dbaToGain(smoothedOutdoorDba), indoor)`, i.e. `dbaToGain` on the *unattenuated* outdoor dBA, then a direct linear multiplier (`INDOOR_SOUND_GAIN_MULTIPLIER = 10**(-35/20)`) applied afterward. This avoids a bug where subtracting 35dB and re-running it through `dbaToGain`'s floor/ceiling clamp silently produced *zero* audible change whenever the outdoor level was already near the audibility floor (common at real Katrineholm-to-turbine GPS distances) — both "ute" and "inne" clamped to the same near-zero gain. The multiplier approach guarantees a consistent, always-perceptible ~35dB-equivalent drop regardless of the absolute outdoor level.
- "🌗 Skuggflimmer" (shadow flicker) mode simulating blade-shadow flicker, only active in "Aktuell sol"/"Låg sol" sun modes, with an info tooltip explaining the effect.
- "📸 Fotomontage": captures a composite photo (camera + AR overlay + watermark + disclaimer) with Spara/Ta ny bild/Dela (Web Share API) actions.
- "🌬️ Infraljud"-monitor: always-visible green/yellow/red badge + expandable panel estimating overall noise/infrasound impact (a distinct indicator from the dBA panel above) from distance, number of contributing turbines, wind direction (if available), and how long the user has been on-site, with a calm Swedish disclaimer emphasizing it is an estimated indicator, not a medical measurement (never claims guaranteed harm).
- Map view (SVG-based) showing all turbines and the user's position.
- "Skriv under för folkomröstning" petition button/modal referencing the real 2022 Katrineholm wind-power referendum.
- Turbines only render in the AR view when the weighted Outdoor Confidence Index (camera/AI sky detection, GPS accuracy, ambient light, compass stability, motion/gyro, WiFi/indoor signal) is in its "show" or "cautious" tier AND camera-based sky coverage is ≥15% of the frame; the "aim" tier (40-70%) shows a "Rikta kameran mot öppen himmel" banner instead of turbines, and the "hide" tier (<40%) or insufficient sky coverage shows the large, high-z-index "Gå utomhus" (go outside) message that always renders above all UI chrome (top/bottom bars, badges). This whole-screen OCI-based gate only ever applies while the (now-removed) ML sky-segmentation model would have been active (`useSkyDetection`'s `method === "ml"`, permanently `"disabled"`) — it never suppresses turbines or shows the "Gå utomhus"/aim banner on its own.
- Per-pixel occlusion (turbines hidden behind trees/buildings, not just a whole-turbine show/hide) is driven separately by `useSkyDetection`'s existing lightweight canvas-2D brightness/texture/saturation heuristic (the same one behind `outdoorConfidence`) — NOT by the old ML model. `isPointSky`/`getOcclusionGrid` now expose the real (EMA-smoothed) 12x8 sky/non-sky grid from that heuristic to `ARScene.tsx`'s existing per-fragment occlusion shader, so only the covered part of a turbine (e.g. below a treeline) is masked. This is coarser than the removed ML segmentation but uses no extra WebGL context and cannot reintroduce the freeze (see Gotchas).
- "🗺️ Placera vindkraftverken själv" (`/placera`), the free "kartverktyg": a pannable/zoomable satellite map (no camera/GPS/compass permissions needed), pre-loaded with the 8 real planned turbines nearest Katrineholm and an initial view wide enough to show Katrineholm/Ericsberg/Forssjö/Björkvik together. The user can click empty space to add a turbine, or tap an existing one to open a context menu (Flytta/Ta bort/Information/Avbryt) — "Flytta" enters a move-mode where the next tap commits the new position, replacing the old drag-to-move gesture so plain drag/pinch/double-tap always pan/zoom the map (Google-Maps-style) when nothing is selected. Each turbine is colored green/yellow/orange/red by a live per-turbine impact score; moving/adding/removing a turbine shows a brief "Beräknar påverkan…" indicator before the score/colors update. A "Visa Ericsbergs mark" toggle overlays a wide illustrative dark-fill/black-border polygon of Ericsberg's estate land, and a 🏰 "Ericsbergs slott" marker shows an owner-info popup. Also has a live overall 0–100 impact score (distance to households, sensitive zones, spacing), factoring in kommun-wide population/household data so placements near central Katrineholm/Forssjö correctly show non-zero household/noise/nature/culture/water impact instead of only scoring against the Ericsberg estate area, playful Swedish warnings, save/compare of multiple placements, and a "Se denna placering i AR" button that swaps the AR view's turbine set to the custom placement. A "✏️ Redigera gräns" mode turns the placement-area boundary into a real editable polygon — drag vertices, add new ones by tapping the map, remove with a ✕, save/reset the custom boundary, and export it as GeoJSON. An expanded "🐞 Felsökning" debug panel shows loaded data-layer counts and a full per-turbine breakdown (households within 1/2/3 km, distance to centrum, nearest nature/cultural/water zone, noise dBA, shadow-flicker score) for diagnosing any factor.
- After pressing the start screen's primary action, the user chooses between "📷 Starta AR" (the full camera/GPS/compass AR flow) or "🗺️ Öppna kartverktyg" (the permission-free map tool above) — both reachable directly from `PermissionGate.tsx`.
- Fully Swedish-language UI; installable as a PWA.

## Gotchas

- iOS Safari requires a user-gesture-triggered `DeviceOrientationEvent.requestPermission()` call before compass data is available — handled in `useCompassHeading.ts`, triggered from the "Starta AR-vyn" button.
- Camera/GPS/compass are unavailable or limited in desktop/headless test browsers — the app's top/bottom UI chrome and map/petition overlays are intentionally still usable even if AR itself can't fully initialize. The dBA and Infraljud panels are gated behind `ready` (needs GPS + orientation + camera stream all present), so they cannot be opened/expanded in a headless/sensor-less test browser — this is expected, not a bug.
- `getUserMedia()` can hang forever on some real devices (reported by users as an infinite "Startar kameran…" spinner) with neither a resolved stream nor a rejection — `useCameraStream.ts` now has a 15s watchdog timer (same pattern as `useGeolocation.ts`'s 20s GPS watchdog) that forces an error + a "🔄 Försök igen med kameran" retry button in `Home.tsx`'s `!ready` overlay if the camera hasn't started by then.
- `LoadingSequence.tsx`'s completion effect previously never fired in practice: `Home.tsx` passed it an inline `onComplete={() => ...}` prop, a new function identity on every re-render (which happens very frequently — sky detection rAF, orientation ticks, GPS), and the effect depended on `onComplete`, so its 500ms completion timer kept getting cleared and restarted forever. The checklist visually reached 100% but never unmounted. Fixed by wrapping the callback in `useCallback` in `Home.tsx`. If the loading UI ever appears to "hang" again after a refactor, check for a freshly-recreated callback/prop identity feeding a timer-based completion effect first.
- The "kartverktyg" (`/placera`) reuses the same real Esri World Imagery satellite basemap already used by `MapView.tsx` (via `webMercatorTiles.ts`) for pan/zoom, rather than introducing Mapbox GL JS — this was a deliberate choice to avoid a new heavy dependency and keep the two maps visually consistent; it does not use the `MAPBOX_TOKEN` secret.
- `PlacementMap.tsx`'s pan/zoom wheel handling uses a native `wheel` event listener added via `useEffect` with `{ passive: false }`, not React's `onWheel` prop — React's synthetic wheel handler is passive by default, so calling `preventDefault()` inside it only logs a browser warning and does not actually block page scroll.
- `useSkyDetection.ts`'s `classifyCell` brightness/texture thresholds are coupled to `GRID_ROWS`/`GRID_COLS`: raising grid resolution shrinks how much of the video each cell downsamples, preserving more real noise per cell. A same-session bump from 12x8 to 16x20 (for treeline precision) left the old `stdDev`/brightness thresholds too strict, misclassifying real open sky (esp. hazy/overcast, near the horizon where the turbines sit) as "occluded" — this made turbines render with the "Visa dolda"-mode's red dashed occlusion tint almost everywhere, and suppressed the aviation light's opacity (driven by the same sky classification) so it barely showed even when blinking. Fixed by retuning thresholds (bright >110, stdDev <18, sat <0.25) to match the finer grid's lower smoothing ratio. See `.agents/memory/sky-heuristic-and-ml-disable-tuning.md` (#3) for the general lesson.
- `PlaceTurbines.tsx`'s "📤 Exportera GeoJSON" deliberately opens a `Blob` URL via `window.open(url, "_blank")` instead of an `<a download>` link — an `<a download>` silently no-ops on iOS Safari/installed-PWA standalone mode (see `.agents/memory/pdf-download-attribute-ios-pwa.md`), so a new-tab approach was used to keep the OS share/save sheet working everywhere.
- `scorePlacement()` reads the active placement boundary via the module-level `getActiveBoundary()` function, not as a parameter — so any `useMemo`/`useCallback` that calls it must add an explicit extra dependency (e.g. a `boundaryVersion` counter bumped on save/reset) or it will keep returning a stale score after the boundary is edited/saved/reset.

## Wind data sync (api-server + lib/wind-sync)

Beyond this AR artifact, the monorepo also has an `api-server` artifact and a `lib/wind-sync` package that ingest Vindbrukskollen (Swedish wind power register) data into Postgres for the `vindkraft-karta` map artifact.

- `lib/wind-sync` — shared lib exporting `runWindSync()` (adapters + upsert logic + locality impact scoring). Used by both the manual CLI script and the API server's automatic scheduler.
- `scripts/src/wind-sync/run.ts` — thin CLI wrapper (`pnpm --filter @workspace/scripts run sync:wind`) for one-off manual syncs.
- `artifacts/api-server/src/lib/windSyncScheduler.ts` — in-process scheduler started on server boot. Runs once ~15s after startup, then every `WIND_SYNC_INTERVAL_HOURS` (default 24; set `WIND_SYNC_DISABLE_SCHEDULER=true` to turn it off). Guards against overlapping runs and tracks last-run status. Exposed via `GET /api/wind/sync-status` (`scheduler` field: `enabled`, `intervalHours`, `isRunning`, `lastRunStartedAt/FinishedAt`, `lastRunStatus`, `lastRunError`, `nextRunAt`).
- The sync is upsert-based on stable external IDs, so re-running it at any time is safe — it only refreshes existing rows and adds newly discovered localities/turbines/project areas.
- The in-process scheduler only keeps data fresh while the api-server process itself is running. If that artifact is ever switched to an autoscale (scale-to-zero) production deployment, pair it with a Replit **Scheduled Deployment** (configured by the user in the Deployments pane — deployment type can't be changed from code) running `pnpm --filter @workspace/scripts run sync:wind` on a cron schedule for a guaranteed periodic run.
