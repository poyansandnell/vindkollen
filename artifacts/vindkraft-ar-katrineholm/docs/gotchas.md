# Gotchas

Detailed pitfalls and fixes for `artifacts/vindkraft-ar-katrineholm`. See `replit.md` for the project overview.

- iOS Safari requires a user-gesture-triggered `DeviceOrientationEvent.requestPermission()` call before compass data is available — handled in `useCompassHeading.ts`, triggered from the "Starta AR-vyn" button.
- Camera/GPS/compass are unavailable or limited in desktop/headless test browsers — the app's top/bottom UI chrome and map/petition overlays are intentionally still usable even if AR itself can't fully initialize. The dBA and Infraljud panels are gated behind `ready` (needs GPS + orientation + camera stream all present), so they cannot be opened/expanded in a headless/sensor-less test browser — this is expected, not a bug.
- `getUserMedia()` can hang forever on some real devices (reported by users as an infinite "Startar kameran…" spinner) with neither a resolved stream nor a rejection — `useCameraStream.ts` now has a 15s watchdog timer (same pattern as `useGeolocation.ts`'s 20s GPS watchdog) that forces an error + a "🔄 Försök igen med kameran" retry button in `Home.tsx`'s `!ready` overlay if the camera hasn't started by then.
- `LoadingSequence.tsx`'s completion effect previously never fired in practice: `Home.tsx` passed it an inline `onComplete={() => ...}` prop, a new function identity on every re-render (which happens very frequently — sky detection rAF, orientation ticks, GPS), and the effect depended on `onComplete`, so its 500ms completion timer kept getting cleared and restarted forever. The checklist visually reached 100% but never unmounted. Fixed by wrapping the callback in `useCallback` in `Home.tsx`. If the loading UI ever appears to "hang" again after a refactor, check for a freshly-recreated callback/prop identity feeding a timer-based completion effect first.
- The "kartverktyg" (`/placera`) reuses the same real Esri World Imagery satellite basemap already used by `MapView.tsx` (via `webMercatorTiles.ts`) for pan/zoom, rather than introducing Mapbox GL JS — this was a deliberate choice to avoid a new heavy dependency and keep the two maps visually consistent; it does not use the `MAPBOX_TOKEN` secret.
- `PlacementMap.tsx`'s pan/zoom wheel handling uses a native `wheel` event listener added via `useEffect` with `{ passive: false }`, not React's `onWheel` prop — React's synthetic wheel handler is passive by default, so calling `preventDefault()` inside it only logs a browser warning and does not actually block page scroll.
- `useSkyDetection.ts`'s `classifyCell` brightness/texture thresholds are coupled to `GRID_ROWS`/`GRID_COLS`: raising grid resolution shrinks how much of the video each cell downsamples, preserving more real noise per cell. A same-session bump from 12x8 to 16x20 (for treeline precision) left the old `stdDev`/brightness thresholds too strict, misclassifying real open sky (esp. hazy/overcast, near the horizon where the turbines sit) as "occluded" — this made turbines render with the "Visa dolda"-mode's red dashed occlusion tint almost everywhere, and suppressed the aviation light's opacity (driven by the same sky classification) so it barely showed even when blinking. Fixed by retuning thresholds (bright >110, stdDev <18, sat <0.25) to match the finer grid's lower smoothing ratio. See `.agents/memory/sky-heuristic-and-ml-disable-tuning.md` (#3) for the general lesson.
- `PlaceTurbines.tsx`'s "📤 Exportera GeoJSON" deliberately opens a `Blob` URL via `window.open(url, "_blank")` instead of an `<a download>` link — an `<a download>` silently no-ops on iOS Safari/installed-PWA standalone mode (see `.agents/memory/pdf-download-attribute-ios-pwa.md`), so a new-tab approach was used to keep the OS share/save sheet working everywhere.
- `scorePlacement()` reads the active placement boundary via the module-level `getActiveBoundary()` function, not as a parameter — so any `useMemo`/`useCallback` that calls it must add an explicit extra dependency (e.g. a `boundaryVersion` counter bumped on save/reset) or it will keep returning a stale score after the boundary is edited/saved/reset.
- (Historical, fixed July 2026) The indoor/no-line-of-sight state used to be shown via a full-screen `z-40` overlay (`inset-0`, `bg-black/85`) in `Home.tsx` — `NearestTurbineArrow` had to render at `z-50` (higher) or it silently disappeared behind that overlay exactly when it was most needed. That overlay is now removed entirely (see the "turbines must never fully disappear" entry below); the indoor state is a small non-blocking `statusBanner` pill instead, so the z-index ordering concern no longer applies to it — but `NearestTurbineArrow` should still stay above any future full-screen overlay if one is ever reintroduced.
- **Safety-fallback vs. "hide when indoors/no-sight" priority inversion (fixed July 2026):** `ARScene.tsx`'s `applyFinalOpacities` used to compute final turbine opacity as `hideAll ? 0 : globalVisibilityFactor`, and `Home.tsx`'s full-screen indoor overlay covered the whole AR view whenever `indoorsOrNoSight` was true — both of these could force opacity to exactly 0 or block the view entirely, which **beat** the 2-second `forceVisible` safety-net fallback (whose entire purpose is to guarantee something is visible). A user report of "no turbines render at all" traced back to this: any of several independent "hide everything" paths (indoor heuristic, low camera-sky confidence, calibration-not-yet-settled) could zero out visibility, and none of them checked whether the safety fallback had already fired. Fixed by making `obj.forceVisible` the highest-priority branch everywhere opacity/tint is computed (both the material opacity in `applyFinalOpacities` and the parallel shadow-opacity calc in `animate()`), replacing hard `0`s with floors (`INDOOR_DIM_FACTOR`, `MIN_CONFIDENCE_VISIBILITY_FACTOR`) so turbines dim to 40–60% + a gray/blue tint instead of disappearing, and removing the full-screen indoor overlay in favor of a non-blocking status pill. **Lesson: whenever multiple independent conditions can each drive a value to a "fully hidden" state, audit that a safety/fallback flag is checked FIRST in every one of them — a fallback that any single hide-path can override is not actually a fallback.**
- No `console.log`/debug output existed anywhere in this artifact before July 2026 — a testable bug report ("nothing renders") had no way to distinguish "GPS never arrived" from "compass never arrived" from "turbines created but forced invisible" from a screenshot alone. Pipeline breadcrumbs (`console.info("[AR][pipeline] ...")` for GPS OK / Compass OK / Loaded N turbines / Creating turbine #i / Anchor created / Model loaded / Renderer attached / Visible=true, plus `console.warn("[AR][safety] ...")` for the 2s force-show fallback) now exist across `Home.tsx` and `ARScene.tsx`, each gated by a `useRef` flag so they log once per session instead of flooding on every re-render/animation frame.
- `ready` (GPS+compass+camera fix) and `LoadingSequence`'s own decoupled internal timeline (calibration → countdown → checklist) run in parallel, so `ready` can flip true while the loading screen is still visually showing — this let the AR scene, HUD (top/bottom bars), arrow, badges, and toasts bleed through on top of/behind the loading screen at startup, since several of those elements also sit at z-index ≥ `LoadingSequence`'s `z-40`. Fixed by adding `arSessionVisible = ready && !showLoadingSequence` in `Home.tsx` and gating every AR-session-only element (ARScene, arrow, night overlay, indoor overlay, calibration/photo-error banners, top bar, bottom bar) behind it instead of `ready` alone. If new HUD/overlay elements are added, gate them behind `arSessionVisible`, not `ready`, unless they're intentionally meant to show during loading.
- **"Verk renderas aldrig trots frisk GPS/kompass/AR-stabilitet" — himmelsmaskens `discard` kunde dölja ALLA verk samtidigt (fixed July 2026):** rotorsaken var INTE i GPS/kompass/stabilitets-pipelinen (som var helt frisk) utan i `ARScene.tsx`s per-pixel-ocklusionsshader: den körde `discard` (fullständigt bortskuret fragment) när `visMask` (himmel-klassificeringen från `useSkyDetection.ts`s grova ljusstyrke/textur/mättnad-heuristik, `classifyCell`) var nära 0. Den heuristiken kan (och gör, i disigt/mulet ljus, motljus, eller när kameran råkar peka mot en ljus men "texturerad" himmel) felklassificera FRI himmel som "ej himmel" över hela eller stora delar av kamerabilden **samtidigt** — vilket lät `discard` slå till för alla 29 verkens samtliga fragment på en gång, trots att de existerade i scenen med korrekt world-position och skala. Fixat genom att ersätta `discard` med ett golv (`OCCLUSION_MIN_ALPHA = 0.18`, `gl_FragColor.a *= mix(OCCLUSION_MIN_ALPHA, 1.0, visMask)`), i linje med samma "dämpa, aldrig ta bort helt"-mönster som `INDOOR_DIM_FACTOR`/`MIN_CONFIDENCE_VISIBILITY_FACTOR`. Se `.agents/memory/occlusion-discard-vs-dim.md`.
- **Felsökningsloggar för "renderas inte"-buggar (juli 2026, tredje omgången):** utöver de befintliga `[AR][pipeline]`-brödsmulorna finns nu de EXAKTA loggraderna `"[AR] GPS position mottagen"` (i `Home.tsx`s GPS-effekt), `"[AR] Modell placerad"` (i `ARScene.tsx`s `layoutObjects()`, en gång per verk när `obj.group.position` faktiskt sätts) och `"[AR] Modell synlig"` (i `animate()`-loopen, en gång per verk när dess faktiska materialopacitet först passerar en synlighetströskel). Dessutom dumpas ett `console.table` var 3:e sekund i `animate()` med per-verk avstånd/bäring/`isVisible`/`frustumVisible` plus kamerans faktiska `cameraForward`-vektor — avsett för att i framtiden på sekunder kunna se OM och VAR i pipelinen (placering vs. synlighet vs. frustum) ett verk tappas bort, istället för att gissa utifrån skärmdumpar.
- **Nearest-turbine-pilen uppdateras redan varje `requestAnimationFrame`** (`NearestTurbineArrow.tsx`s `tick()`-loop läser `getCurrentHeading()` — samma delade funktion som AR-scenens bäringsjämförelser — varje bildruta, oavsett om `bearingDeg`-proppen från React-state ändras eller ej). En rapport om att pilen "känns statisk" pekar troligen mot att kompassen själv inte uppdateras (t.ex. `useDeviceOrientation.ts`s frusna-värden-fall, se posten nedan), inte mot pil-komponentens egen uppdateringslogik — kontrollera `SensorDebugPanel`s "Heading updates/sec" innan du misstänker pilen.
- **"Synliga verk: 0" trots frisk GPS/kompass/world-update — kamerans `matrixWorldInverse` var en bildruta gammal (fixed July 2026, tredje rapporten):** varken position, skala eller synlighetsflaggor var fel — problemet var att `animate()` satte `state.camera.quaternion.copy(...)` men aldrig anropade `state.camera.updateMatrixWorld()` direkt efter. Three.js/`@react-three/fiber` uppdaterar normalt `matrixWorldInverse` (som frustum-testet och alla kamera-relativa bäringsberäkningar läser) inne i `renderer.render()`, vilket körs i SLUTET av bildrutan — så all frustum-/synlighetslogik som körde TIDIGARE i samma `animate()`-anrop läste föregående bildrutas kameraorientering. Med en snabbt roterande telefon (kompassen uppdateras varje bildruta) kunde detta systematiskt hålla varje verk "utanför frustum" enligt det förra läget, medan det faktiskt var synligt i det nuvarande. Fixat med ett explicit `state.camera.updateMatrixWorld(true)` omedelbart efter quaternion-tilldelningen, innan något frustum-/synlighetstest görs. **Lesson: en kameras `matrixWorldInverse`/`projectionMatrix` är INTE garanterat uppdaterad bara för att `quaternion`/`position` precis sattes — om egen kod läser projicerade/kameraspace-koordinater FÖRE `renderer.render()` i samma bildruta, tvinga fram `updateMatrixWorld()` manuellt.**
- **Felsökningsverktyg för "verk renderas inte"-rapporter (juli 2026, tredje omgången):** `SensorDebugPanel` har nu en "🛠️ Felsökningslägen"-sektion med två oberoende omkopplare (helt fristående från alla produktinställningar som "Visa dolda verk"): (1) **"Tvinga markör på närmaste verk"** (`debugForceNearest`) ritar en gul vertikal linje (mark→nav) + ett rött klot vid navet + en textetikett (GPS/ENU/bäring/avstånd) på det GPS-mässigt närmaste verkets FAKTISKA världsposition, varje bildruta, helt okopplat från ocklusion/himmelsmask/frustum/AI-segmentering — om markören syns men verket självt inte gör det är felet i synlighets-/ocklusionspipelinen, inte i placeringen. (2) **"Stäng av ocklusion/AI-segmentering"** (`disableOcclusion`) kortsluter både `applyFinalOpacities`s dämpningsfaktorer och shaderns `uShowHidden`-uniform till 1 samtidigt, för att isolera om ett "syns inte"-fel kommer från ocklusion/himmelsdetektering eller från något annat (frustum, skala, position). Det utökade `console.table` (var 3:e sekund) innehåller nu även lat/lon, ENU x/y/z, bäring, pitch, `inFrustum`, `occluded`, `isVisible`, `screenX`/`screenY` per verk för att kunna se exakt VAR i pipelinen (position → frustum → ocklusion → slutlig opacitet) ett verk faller bort.
- **Kamera-/GPS-beroende AR-flöden kan inte E2E-testas i sandboxad testbrowser:** testagentens Playwright-browser saknar en riktig kamera och nekar/misslyckas `getUserMedia()` även med `camera`-permission beviljad (visar "Ingen kamera hittades på enheten."-felskärmen permanent) — detta blockerar allt UI bortom `PermissionGate`, inklusive `SensorDebugPanel` och de nya felsökningsomkopplarna ovan. Verifiera sådana ändringar via typecheck + manuell kodgranskning av rit-/tillstånds-logiken istället för `runTest()`; se även `.agents/memory/sandbox-webgl-unavailable.md` för den relaterade WebGL-begränsningen.
- **"Verken hänger kvar på skärmen vid en snabb ~90°-vridning" (fjärde kritiska buggrapporten, fixed July 2026):** kamerans rotation VAR redan korrekt implementerad i world-space (fräscht `state.camera.quaternion.copy(quaternionRef.current)` + `updateMatrixWorld(true)` varje bildruta, se posten ovan om "Synliga verk: 0") — problemet satt uppströms, i `useDeviceOrientation.ts`s adaptiva riktningsutjämning. `HEADING_TURN_CONFIRM_SAMPLES` (2) krävde att TVÅ på varandra följande avläsningar över bruströskeln pekade åt samma håll innan den snabba tidskonstanten (`HEADING_TURN_TAU`) släpptes igenom — ett medvetet skydd mot handskakningsspikar, men det innebar att en enda ovanligt stor engångsdelta (t.ex. efter en tillfällig sensor-eventpaus) tvingade igenom en extra väntecykel innan kameran svarade snabbt, vilket på enheter/OS-versioner med glesare eventtakt kunde kännas som att objekten "hänger kvar" i över en sekund efter en snabb vridning. Fixat med en "escape hatch" (`HEADING_LARGE_JUMP_DEG = 20`): en delta så stor kan per definition inte vara magnetometerbrus, så den räknas som direkt bekräftad utan att vänta på en andra avläsning. Samtidigt utökades felsökningsloggen (`console.table`/`console.info` i `ARScene.tsx`s `animate()`) med explicita `relativeAngle_deg`-fält per verk och en global `Heading=...°  Camera yaw=...°`-rad — `Camera yaw` läses direkt ur `state.camera.quaternion` (samma bildruta, efter tillämpning), så framtida divergens mellan visad `Heading` och faktisk `Camera yaw` blir omedelbart synlig i loggen istället för att behöva gissas fram.
- **`LiveDebugStrip` överlappade permanent "Katrineholm · N verk"-titeln (fjärde kritiska buggrapporten, fixed July 2026):** `Home.tsx`s `topBarHeight`-mätningseffekt (som placerar felsökningsraden precis under topp-baren via `ResizeObserver`) berodde på `[started]`, men `topBarRef`s `<div>` renderas villkorat på `ready`/`arSessionVisible` — vilket alltid blir sant EFTER `started` (GPS/kompass/kamera-fix tar sekunder). Effekten kördes alltså EN gång redan vid `started`, hittade `topBarRef.current === null` (elementet fanns ännu inte i DOM:en), satte `topBarHeight` till 0 och kördes sedan ALDRIG om — så remsan trodde permanent att topp-baren hade höjd 0 och ritades alltid på `top: 0`, rakt ovanpå titeln. Fixat genom att bero effekten på `ready` istället (samma flagga som faktiskt styr om elementet finns i DOM:en). **Lesson: ett `useEffect` som mäter/observerar ett DOM-element via en `ref` måste bero på EXAKT den flagga som styr om det elementet är monterat — inte på en tidigare flagga i samma laddningskedja — annars kan effekten köra en gång "för tidigt", hitta `ref.current === null`, och aldrig återhämta sig.**
- **"Direkt AR" → "World locked"-övertoning, ALDRIG blockerande ocklusion (SJÄTTE kritiska buggrapporten, fixed July 2026):** produktkravet var att verk ALLTID renderas direkt vid AR-start (även inomhus, ingen ocklusion/himmelsmask/inomhus-heuristik får dölja dem initialt), och att övergången till full GPS/kompass/pitch-stabiliserad rendering 5s senare aldrig känns som ett hopp. Löst i `ARScene.tsx` med `WORLD_LOCK_BLEND_MS = 5000` och en delad `worldLockBlendRef` (0 vid `arStartedAtMs`, linjärt mot 1 över 5s) som blandas in på TVÅ separata ställen som annars kunde dämpa/dölja verk: (1) `attachOcclusionShader`s fragment-shader (`uWorldLockBlend`-uniform, `gl_FragColor.a *= mix(1.0, occlusionAlphaFactor, uWorldLockBlend)` — vid blend=0 tvingas ocklusionsalpha till 1.0 oavsett himmelsmask) och (2) `applyFinalOpacities` (blandar in de "naturliga" — dvs. redan beräknade — global-/sky-faktorerna via `1 + (natural-1)*worldLockBlend`, istället för att hoppa direkt till dem). Samma mönster: identifiera VARJE plats i koden som kan sänka en verk-opacitet, och blanda in en delad 0..1-blend-faktor på alla samtidigt — annars kan en enda missad plats fortfarande dölja verk under "Direkt AR"-fönstret. Den separata "Visa/dölj verk"-togglen (`turbinesVisible`-prop, `TURBINES_VISIBLE_FADE_MS = 500`) är en HELT oberoende blend-faktor (`turbinesVisibleFactorRef`, tidsbaserad lerp mot mål 0/1) som multipliceras in separat — den får aldrig blandas ihop med `worldLockBlendRef`, eftersom den ena är sensor-/tids-härledd och den andra ett rent manuellt användarval.
- **"Kompassen fryser helt" — total value-freeze gap (fixed July 2026):** `useDeviceOrientation.ts` already had two frozen-sensor safeguards — a per-event "heading stuck while pitch/roll still moving" fallback (`HEADING_STALE_MS`, falls back to raw `alpha`) and a watchdog for *total* event silence (`ORIENTATION_STALLED_MS`, reattaches listeners). Neither caught the case a user actually reported (arrow/turbines frozen in place while turning the phone): the browser keeps firing `deviceorientation(absolute)` events at a normal rate (so the silence watchdog never trips), but alpha/beta/gamma are all bit-for-bit identical every time (so the "pitch/roll still moving" gate on the per-event fallback never fires either) — a known stuck-internal-fusion state on some Android browsers. Fixed by adding a third, independent check (`STUCK_VALUES_MS` = 1800ms): if *neither* raw heading *nor* raw pitch/roll has changed for that long despite events still arriving, the watchdog treats it exactly like total silence (zeroes stability refs, tears down + re-adds the listeners) — the `event.alpha` fallback can't help here since it's frozen too, so listener reattachment is the only remedy. Also added `updatesPerSecond`/`lastUpdateAgeMs`/`valuesFrozen` fields (computed once per watchdog tick, not per event, to avoid a 15-60Hz re-render) surfaced in `SensorDebugPanel.tsx` as "Heading updates/sec" and "Last update" so a future field report can distinguish "no events at all" from "events but stuck values" from "genuinely no motion" at a glance.
- **"Verken uppdateras trögt (~1-2Hz), hänger ~1s, hoppar sedan" — SJUNDE kritiska buggrapporten, punkt 1 (fixed July 2026):** `ARScene.tsx`s `animate()`-loop rekomputerade redan `layoutObjects()`/kameraquaternionen varje obromsad `requestAnimationFrame`, så render-loopen var aldrig boven. Roten satt i `useDeviceOrientation.ts`s riktningsutjämning: `HEADING_STILL_TAU`/`PITCH_ROLL_STILL_TAU` (tidskonstanterna som gäller när ingen "riktig vridning" ännu bekräftats) var satta konservativt (1.3s/0.9s) för att undertrycka magnetometerbrus i stillhet — vilket gav upplevelsen av trögt/hackigt släpande verk under en pågående vridning innan `HEADING_TURN_TAU`s snabbare läge hann kopplas in. Fixat med riktig sensorfusion (se punkt 3-posten nedan): `handleMotion` läser `devicemotion`s `rotationRate`-MAGNITUD (aldrig tecken/riktning — se motiveringen där) och en `GYRO_TURN_RATE_THRESHOLD_DEG_PER_SEC`-tröskel för att avgöra "enheten vrids verkligen just nu", vilket kortsluter både `HEADING_TURN_CONFIRM_SAMPLES`-väntan (samma "escape hatch"-mönster som den befintliga `HEADING_LARGE_JUMP_DEG`, se posten om fjärde buggrapporten ovan) OCH de nu snålare still-tidskonstanterna, samtidigt.
- **"Skärmdump väcker plötsligt verken" — render-loop kunde fastna helt tyst innan första bildrutan (SJUNDE buggrapporten, punkt 2, fixed July 2026):** `lastFrameAtRef` initierades till `null` och sattes bara INIFRÅN `animate()` självt — så om webbläsaren av någon anledning aldrig levererade den allra första `requestAnimationFrame`-callbacken (t.ex. en flik öppnad i bakgrunden, eller GPU-drivrutinen som tillfälligt inte schemalägger rAF), stod `lastAt === null` kvar för alltid och den fristående 250ms-vakthunden (som explicit `return`:ar på `null`, se posten om "pilen/verken fryser helt" ovan) fick ALDRIG en chans att riva/återskapa loopen. Fixat genom att sätta `lastFrameAtRef.current = Date.now()` direkt INNAN den första `requestAnimationFrame(animate)`-anropet görs — inte bara inifrån `animate` — så vakthunden alltid har ett konkret jämförelsevärde. Samtidigt tillkom explicita `console.info`-loggar för `"[AR][pipeline] startAR() anropad (knapptryckning)"` (Home.tsx, i knappens `onClick`), `"[AR][pipeline] startAR(): render-loop initierad, väntar på första renderFrame()"` (ARScene.tsx, precis innan den initiala `requestAnimationFrame`) och `"[AR][pipeline] Första renderFrame() kördes"` (en gång, inuti `animate()`s första faktiska anrop) — så en framtida "verken dyker inte upp förrän X händer"-rapport direkt kan avgöra om `animate()` överhuvudtaget någonsin kördes.
- **Riktig sensorfusion (gyroskop för snabba vridningar, kompass bara för långsam drift-korrigering) — SJUNDE kritiska buggrapporten, punkt 3 (fixed July 2026):** debug-panelens "Källa: kompass" speglade korrekt att headingVÄRDET alltid kommer från kompassen (`alpha`/`webkitCompassHeading`) — men användaren ville att gyroskopet faktiskt skulle DRIVA snabba riktningsändringar, med magnetometern bara för långsam drift. Löst UTAN att integrera rå gyro-vinkelhastighet till ett absolut headingvärde (medvetet undvikit — tecken-/axelkonventionen för `rotationRate.alpha/beta/gamma` skiljer sig mellan enheter/webbläsare och går inte att verifiera utan fysisk enhet, se `.agents/memory/`-posten om detta): en ny `devicemotion`-lyssnare (`handleMotion`) läser bara MAGNITUDEN av rotationshastighetsvektorn och jämför mot `GYRO_TURN_RATE_THRESHOLD_DEG_PER_SEC` (12°/s). När den passeras sätts `gyroConfirmsRealMotion`/`motionFusionActive` = true, vilket (a) kortsluter `HEADING_TURN_CONFIRM_SAMPLES`-väntan precis som en stor engångsdelta redan gjorde, och (b) väljer den snabba `HEADING_TURN_TAU`/tilt-motsvarigheten istället för de (nu snålare) still-konstanterna — så gyroskopet agerar "vridning pågår verkligen"-bekräftelse/förstärkning för SNABBHETEN i utjämningen, medan kompassen förblir den enda källan till det FAKTISKA headingvärdet. `motionFusionActive` exponeras i `LiveDebugStrip` som ett separat "Sensorfusion: gyro+kompass / endast kompass"-fält, fristående från det äldre "Källa: kompass/gyro"-fältet (som är ett helt annat, redan existerande fryst-fallback-koncept). `DeviceMotionEvent.requestPermission()` (iOS 13+) triggas synkront bredvid `DeviceOrientationEvent.requestPermission()`, inte efter ett `await`, av samma skäl som beskrivs i `.agents/memory/user-gesture-permission-chaining.md`.
